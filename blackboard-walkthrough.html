SECTION 1 — Project Identity

1.1 — What is BlackBoard?

BlackBoard is a hybrid collaborative workspace that combines a text editor and a shared whiteboard into a single accessible web app. It is designed for quick group thinking, ad hoc planning, and lightweight collaboration without forcing users through heavy account creation or first-time onboarding.

The product solves the problem of scattered collaboration tools: users should not have to switch between a note document, a drawing surface, and a chat window to sketch ideas. Instead, BlackBoard gives students, educators, solo creators, and small teams a unified destination for structured writing and freehand visual thinking.

The target user is a learner or sharer who wants immediate frictionless collaboration: a study group sketching a concept map, a teacher distributing a timed interactive board, or a small team exporting a rough diagram. It fills a gap between tools such as Figma, Notion, Miro, and Google Docs by prioritizing instant link-based access, simple interface, and a lightweight web deployment.

This project exists because real-time collaboration is most valuable when it is easy to start. BlackBoard trades enterprise complexity for a simple, fast shared canvas that can be created and joined with a single link.

1.2 — The Core Value Proposition

Real-time collaboration without sign-up friction. Users join by entering a display name stored in session storage, and the app uses temporary JWT authentication for board-level sockets rather than forcing full account creation.

Unified workspace: whiteboard + text editor in one place. The board page combines collaborative text editing and a shared whiteboard tab, so teams can brainstorm visually and capture notes without swapping apps.

Lightweight deployment. The project uses a React single-page frontend paired with an Express API backend and MongoDB persistence. It intentionally avoids Electron, desktop packaging, and heavy multi-service orchestration.

Focused product scope. The platform is intentionally narrow: it is a collaborative whiteboard built for easy link sharing and session-based writing, not a full productivity suite with calendar syncing or enterprise identity management.

SECTION 2 — Tech Stack Deep Dive

2.1 — Frontend

The frontend is built with React 18 and Vite, not Create React App. Vite provides near-instant hot module replacement, faster cold starts, and a modern ES module-based dev server — critical benefits when iterating on a complex canvas application.

React 18 (Vite)

React is the UI library powering BlackBoard's interactive pages. React 18 adds concurrent rendering and automatic batching of state updates, which helps with the many rapid state changes that happen during whiteboard interactions (drawing, dragging, syncing). The declarative component model maps naturally to the overlay-based object system.

Vite was chosen over Create React App because CRA uses Webpack under the hood and has a slow dev server on large projects. Vite compiles only what's needed using native ESM, making the dev loop noticeably faster. Alternatives like Next.js were rejected because BlackBoard is a pure SPA — there's no need for server-side rendering or file-based routing.

React Router v6

React Router v6 enables client-side navigation between the dashboard, all-boards page, starred, trash, individual board/whiteboard pages, and the admin panel — all without full page reloads.

v6 was chosen because it is the current stable version with a cleaner nested route API. The older v5 API required more boilerplate for nested layouts. A flat custom router was rejected because React Router's declarative `<Routes>` and `<Outlet>` make the DashboardLayout shell pattern clean and maintainable.

Lucide React

Lucide React provides lightweight SVG icons for the toolbar, sidebar navigation, and object UI. The repo imports icons such as Pen, Eraser, Trash2, Square, StickyNote, Star, Share2, and ChevronLeft.

Lucide was chosen over FontAwesome or Material Icons because it is fully tree-shakable (you only bundle what you import), MIT licensed, and easy to style with CSS color and size props. It keeps the bundle lean compared to icon fonts which load entire font files.

TipTap (Rich Text Editor)

TipTap powers the rich text editor inside the Board textboard tab. It is built on ProseMirror and provides a modular extension system. BlackBoard uses: StarterKit, Placeholder, Typography, Color, TextStyle, Highlight, TaskList, TaskItem, Image, and Link extensions.

TipTap was chosen over Quill (too opinionated about DOM structure), Draft.js (Meta-maintained, complex API), and Slate (requires too much custom logic). Critically, TipTap's collaboration extension (Yjs) was explicitly excluded — the project uses debounced JSON sync instead, which is simpler and sufficient for "mostly one editor at a time" usage.

qrcode.react

The Share Modal uses qrcode.react to render a scannable QR code pointing to the board URL. It renders as an SVG with the gold accent color on a transparent background, sized at 160×160px.

qrcode.react was chosen because it is a focused, zero-dependency library. It avoids server-side QR generation and keeps the implementation entirely in the frontend.

react-hot-toast

All user feedback toasts — board created, link copied, saved, moved to trash, restored, starred, connection lost — use react-hot-toast positioned at bottom-right. It replaced every alert() call in the codebase.

react-hot-toast was chosen for its tiny bundle size, simple API (toast.success(), toast.error()), and easy custom styling that matches the BlackBoard dark theme.

Socket.io client

The frontend uses socket.io-client to open a persistent WebSocket connection to the backend. It emits and receives events for stroke sync, object updates, text updates, cursor movement, typing indicators, and presence.

Socket.io was chosen over raw WebSockets because it provides automatic reconnection, room management, event acknowledgements, and a stable API without extra implementation effort. The client is initialized as a singleton in socket.js to prevent duplicate connections on re-renders.

2.2 — Backend

Node.js + Express

The backend is built on Node.js with Express as the HTTP framework. Express provides simple routing, middleware, and JSON body parsing for the API endpoints.

Fastify and NestJS were rejected in favor of Express because the codebase benefits from minimal ceremony and direct access to underlying request/response objects. For a modest API with a small number of endpoints, Express is the right fit.

MongoDB + Mongoose

The server uses MongoDB for persistence and Mongoose for schema modeling. Board documents store text content, whiteboard data, attached image URLs, expiration metadata, and lock state.

PostgreSQL was not chosen because the data model is naturally document-oriented: whiteboard objects are heterogeneous, and flexible arrays are easier to store in MongoDB. MongoDB fits the rapid prototyping and schema flexibility the project requires.

Socket.io server

Socket.io is also used on the backend to accept authenticated websocket connections and relay events. It is configured with CORS and a simple JWT auth gate.

Raw WebSockets were not chosen because Socket.io provides room support, reconnect handling, and acknowledgements out of the box. Pusher would add an external dependency and additional cost for this project.

JWT

JWT is used for socket authentication and admin authentication, not for full end-user identity. For boards and whiteboards the backend issues a JWT token after the client successfully unlocks a board or loads a whiteboard.

On the admin side, `admin/login` returns a token that is stored in localStorage and attached to admin route requests. This is sufficient for the current protected dashboard scope.

nanoid

The backend uses `nanoid` to generate short, URL-safe board IDs for normal boards. It is preferred over UUID because the generated strings are compact, readable, and collision-resistant for the scale of this project.

For standalone whiteboards, the code uses `crypto.randomBytes(4).toString('hex')` instead of `nanoid`, but the same goal is achieved: a short shareable ID.

dotenv

The backend loads secrets from `.env` via `dotenv`. This keeps the connection string, JWT secret, and client URL out of source code and enables different values in development and production.

2.3 — Infrastructure & Deployment

This repository is structured as a monorepo with a backend folder and a frontend folder. In a deployed state, the frontend can be hosted on Vercel, while the backend can run on a service such as Render or Heroku.

Vercel is ideal for the frontend because it auto-deploys from GitHub and serves static files with global CDN caching. The backend requires a platform that supports long-running Node processes and Socket.io connections.

MongoDB Atlas is the cloud database service for this project. The app expects a connection string, an IP whitelist or access rule, and environment variables for credentials. Atlas is chosen because it provides a managed cluster and scales flexibly with the document-oriented data model.

Environment variable strategy

The backend stores secrets in `.env`, including `JWT_SECRET`, `MONGO_URI`, and `CLIENT_URL`. Production values should be configured in the Render dashboard or whichever host is running the backend.

The frontend uses `REACT_APP_API_URL` in its own environment and build-time configuration. It should never store sensitive secrets in the frontend bundle because all frontend code is visible to any user.

In a deployed stack, public values like the API endpoint can live in Vercel environment variables for the frontend, while truly secret values remain on the server only.

Secrets must never be in the frontend bundle. The React app only uses VITE_API_URL (Vite projects prefix env vars with VITE_, not REACT_APP_), which is safe because it points to the backend URL — not a secret. Any JWT secret or database URI belongs exclusively to the backend environment and is never shipped to the browser.

3.2 — Frontend Architecture

The actual folder structure in the frontend is closer to a classic Create React App layout, but the conceptual architecture is still grouped by role.

frontend/src/
  components/
    Shared UI pieces and whiteboard overlay components
  pages/
    Route-level pages such as Landing, CreateBoard, BoardPage, WhiteboardPage
  App.js
  Root routing and theme state

In this repo, `components/` contains reusable UI and object components such as `TextObject`, `ShapeObject`, `StickyNote`, `ImageObject`, `Editor`, and `Whiteboard`. These objects are separated from pages so the whiteboard behavior can be reused in both a standalone whiteboard page and the board page.

The `pages/` folder contains route-level components that manage page-level state and API integration. For example, `BoardPage.jsx` orchestrates board unlocking, socket setup, and tab switching, while `WhiteboardPage.jsx` is the standalone real-time board experience.

There is no dedicated `hooks/`, `context/`, or `utils/` folder in this repository yet; most shared logic is embedded in page and component modules. This is a manageable tradeoff for the current size, but a future refactor would benefit from splitting socket logic into hooks and utilities.

3.3 — Backend Architecture

The backend structure is a clean Express architecture with controllers, routes, models, and middleware.

server/
  models/
    Mongoose schemas: Board.js, Whiteboard.js
  controllers/
    Business logic for board and admin endpoints
  routes/
    Express route definitions for API endpoints
  middleware/
    Auth, error handling, upload utilities
  server.js
    Entry point, Socket.io server, REST mount points

The folder separation is deliberate: models define schema, controllers execute business rules, routes expose endpoints, and middleware is responsible for cross-cutting concerns such as error handling and authentication.

The `server.js` entrypoint initializes Express, configures CORS, mounts routes, and sets up the Socket.io server. Socket.io event handlers live directly in `server.js` for this repository rather than in a separate socket module.

3.4 — The Unified Object System

Every whiteboard overlay object is modeled with a minimal shared shape: an object has an `id`, `type`, `x`, `y`, and optionally width, height, plus type-specific fields. This consistency is critical for render logic, drag behavior, synchronization, and undo/redo.

If objects do not follow the same structure, the code would need special-case branches for selection, serialization, and history management. That leads to duplicated logic and hard-to-debug sync bugs.

The `type` field acts as a discriminator: the renderer chooses different React components or canvas semantics depending on whether the object is `text`, `shape`, `sticky`, or `image`.

Comparison of Object Types

TypeFieldsNotes

shape{ id, type, shapeType, x, y, width, height, color, fillColor, strokeWidth, userName }SVG shape overlay with move/resize handles.
sticky{ id, type, x, y, width, height, content, color, userName }Sticky note card with contentEditable body and palette.

3.5 — Real-Time Sync Architecture

The Socket.io flow in the standalone whiteboard is a simple low-latency relay. When a user draws or updates an object, the client emits an event to the server, the server broadcasts it to the room, and other clients update their local state.

For the normal board page, the same pattern applies to text edits, image uploads, and whiteboard objects.

Event NameDirectionPayloadPurpose
join_whiteboardClient → Server{ whiteboardId, userName }Join standalone whiteboard room.
wb_draw_syncClient → Server{ whiteboardId, stroke }Stream in-progress stroke data to others.
wb_receive_sync_strokeServer → Clients{ stroke }Render live stroke preview from remote users.
wb_draw_strokeClient → Server{ whiteboardId, stroke }Commit finished stroke to room and persist buffer.
wb_receive_strokeServer → Clients{ stroke }Append completed stroke to remote canvas.
wb_image_addedClient → Server{ whiteboardId, image }Broadcast new overlay object.
wb_image_updatedClient → Server{ whiteboardId, image }Update object position/content for everyone.
wb_image_removedClient → Server{ whiteboardId, imageId }Remove overlay object across clients.
wb_undoClient → Server{ whiteboardId }Request undo and broadcast the change.

The server is intentionally a relay. It does not reprocess drawing paths or normalize object coordinates; it forwards events and persists the resulting object arrays. This keeps the backend stateless enough for a single-instance deployment.

Undo/redo synchronization is implemented as a broadcast event: when one client triggers undo, the server emits `wb_undo` and all clients remove the last stroke locally. This is simple but does not guarantee deterministic results under simultaneous undo operations.

Stroke sync is immediate, while rich text and object moves are throttled or debounced in the components. The current implementation uses a 500ms throttle for text object updates and selective immediate sends for final object commits.

3.6 — State Management Philosophy

This app deliberately avoids Redux, Zustand, or other global state libraries. It uses local React state and props to keep components contained, with a small amount of persisted storage for identity and theme.

  • Local component state is used for UI details such as `isDrawing`, `toolType`, `shapeType`, `textInput`, and `activeTextId`.
  • React Context is not heavily used in the repo; theme state is passed by props from `App.js`.
  • localStorage / sessionStorage store the user preference theme and temporary display name for boards.
  • Server state lives in MongoDB: board content, whiteboard strokes, and attached image metadata.

This is appropriate at the current scale because the app has a small number of screens and relatively simple state interactions. It would start to break down when multiple pages require the same complex shared state or when cross-component synchronization becomes brittle.

At larger scale, Redux or Zustand would be justified once the app has significant persistent application state, deeply nested pages, and many simultaneous collaboration contexts.

3.7 — The Coordinate System

Coordinate handling is one of the most delicate parts of the whiteboard experience.

The raw mouse coordinate system is the browser pixel space from the top-left of the canvas element. The canvas coordinate system is also based on the canvas origin, but it may need to account for pan and zoom transforms.

In a zoom-enabled architecture, normalized coordinates are computed as:

const canvasX = (mouseX - panX) / zoom; const canvasY = (mouseY - panY) / zoom;

That normalization must happen everywhere drawing, placing, dragging, or resizing occurs. If you forget to normalize, a sticky placed while zoomed will appear far from the intended location.

For displaying remote cursors, the reverse transform is:

const displayX = x * zoom + panX; const displayY = y * zoom + panY;

This normalization must happen in every placement, drag, and resize handler — drawing, placing shapes/stickies/text, dragging objects, and resize handles. Missing it in even one place causes objects to appear at the wrong position when zoomed. For displaying remote cursors, the reverse transform converts canvas coordinates back to screen space: displayX = x * zoom + panX. This ensures a remote cursor drawn at canvas position (100, 100) appears at the correct screen pixel regardless of the local user's zoom level.

SECTION 4 — Feature-by-Feature Technical Breakdown

4.1 — Pen/Drawing Tool

From the user perspective, the pen tool allows freehand drawing on the canvas with a selectable stroke color and width. The eraser mode also behaves as a special path type.

Technically, the whiteboard uses an HTML `<canvas>` element to render stroke points. The component collects mouse positions during `mousemove` and stores them in `currentStrokeRef.current.points`. On `mouseup`, the stroke is committed into the `strokes` array and emitted to the server.

The hardest part is rendering reliably across high-DPI displays and preserving the drawing while the canvas resizes. The code solves this by scaling the canvas context with `window.devicePixelRatio` and redrawing the stroke history on every resize.

An important edge case is that canvas state is not preserved automatically when the canvas size changes. The implementation takes a snapshot and redraws all strokes to avoid pixelation.

4.2 — Text Objects on Canvas

Text objects appear as editable boxes on the whiteboard. Users can place them with a click, edit the content, and move them with pointer drag.

Text is rendered as a DOM overlay using `contentEditable` instead of being drawn directly onto the canvas. This preserves native text selection, cursor behavior, and line wrapping without the complexity of canvas text metrics.

The attribution label is stored in the object as `authorName` and rendered permanently above the text box so anyone can see who created it.

The repo includes a typing indicator system in concept, but the current implementation mostly uses `text_typing` events for the board text editor rather than live per-text-object typing indicators.

4.3 — Shapes (Rectangle, Circle, Triangle, Diamond)

Shapes are created with mouse drag: the user chooses the shape tool, presses down, drags to size, and releases to commit the final shape.

The implementation uses a ghost preview drawn on the canvas during drag, then creates a shape object with `type: 'shape'` and `shapeType` set to one of `rect`, `circle`, `triangle`, or `diamond`.

Instead of using CSS clip-path for shapes, the repository uses inline SVG inside the overlay component. That guarantees crisp rendering and simple preview behavior.

Each shape is editable via move and resize handles. The active shape shows a toolbar and a resizable corner handle. The hardest part is making the drag and resize feel smooth while keeping the object state consistent.

4.4 — Sticky Notes

Sticky notes are placed by clicking with the sticky tool. Each note is a draggable, resizable card with a color palette and a `contentEditable` text body.

Rotation is intentionally not supported, because any rotation would complicate coordinate math, drag boundaries, and object alignment. The repo does not apply CSS rotate transforms to sticky notes.

The sticky palette includes five colors, and text uses a darker ink color for readability. This is deliberate: a consistent dark text color like `#2a1f0e` is easier to read on pastel sticky backgrounds.

Resize is supported from the bottom-right handle only, which reduces complexity and avoids the need to manage 8-directional resizing math.

4.5 — Undo / Redo

Undo removes the last committed stroke from the canvas and pushes it onto a redo stack. Redo re-applies the last removed stroke.

In the standalone whiteboard, the app tracks `strokes` and `redoStack` as arrays in state. When `undo` is triggered, the client emits `wb_undo` and updates local state. The server broadcasts `wb_undo` so all connected clients remove the last stroke.

This implementation treats undo as a shared command, not a per-client history. That works for simple collaborative sessions, but it breaks when multiple users undo at the same time because the definition of "last stroke" may differ across clients.

The repo currently uses full snapshots of the `strokes` array in memory, not delta patches. This is simpler but means that undo state is kept entirely on the client.

4.6 — Zoom & Pan

Zoom and pan are fully implemented in the whiteboard. A zoom control in the bottom-right corner shows the current zoom level (10% – 400%) with −, +, and Fit buttons. The canvas scales smoothly via CSS transform.

The implementation applies transform: scale(zoom) translate(panX, panY) with transform-origin: 0 0 on the canvas container. The 0 0 origin is critical — using the default center origin would produce incorrect coordinate offsets for all mouse events. Zoom centered on the cursor uses the formula: setPanX(mouseX - newZoom/zoom * (mouseX - panX)), keeping the point under the cursor fixed as zoom changes.

Pan is triggered in three ways: Space + drag (cursor changes to grabbing), middle-mouse-button drag, or the Pan tool (H key). The hardest part was making every single mouse handler apply the same normalization: canvasX = (mouseX - panX) / zoom. Missing this in even one handler causes objects to jump to the wrong position.

The Fit button calculates a bounding box of all objects, then sets zoom and pan to center all content in the viewport — useful after adding many widely-spaced objects.

4.7 — Real-Time Presence (Avatars)

Every connected user appears as a 32px circular avatar in the top bar of every board and whiteboard. Avatars stack with a −8px overlap, show user initials, and display a tooltip on hover with the full username. If more than 4 users are present, a +N overflow pill appears.

The server assigns each user a color deterministically on socket join: it counts existing users in the room and indexes into a 6-color palette (#C9A84C gold, #ED93B1 pink, #AFA9EC lavender, #6EC9A8 mint, #E8956D coral, #64B5F6 sky blue). This assignment is sent back to the client via joined_board event as { userId, userName, color, userNumber }. The same color is used for the attribution label, cursor, and avatar — creating a consistent visual identity per session.

The user_list event is broadcast to the whole room whenever someone joins or leaves, updating the avatar bar in real time. Avatars fade in on join and fade out on leave using a 300ms CSS transition. An online indicator shows ● N online in green to the left of the avatar stack.

4.8 — Cursor Presence

On the whiteboard, every connected user's cursor appears as a small SVG arrow in their assigned color, with a name label pill below it. Remote cursors move smoothly using a 50ms CSS transition and fade out automatically after 4 seconds of inactivity.

The client throttles cursor_move events to exactly 50ms (20fps). The bandwidth math: 20 events/s × ~60 bytes payload = ~1.2 KB/s per user — entirely acceptable. The server relays the event to all other room members without processing it. The client filters out its own cursor by comparing userId so you never see yourself.

Cursor coordinates are stored in canvas space (normalized for zoom), then converted back to screen space for display: displayX = x * zoom + panX. This means remote cursors appear at the correct position regardless of whether the local user has zoomed or panned. The hardest part was ensuring this reverse transform was applied consistently — missing it makes remote cursors appear in the wrong position when zoomed.

The stale cursor cleanup runs on a 1-second interval: any cursor whose lastSeen timestamp is more than 4 seconds old is removed from state. This handles the case where a user's browser tab is closed without a clean socket disconnect.

4.9 — Rich Text Editor (TipTap)

This project currently does not use TipTap. The existing collaborative editor is implemented as a plain `<textarea>` in `Editor.jsx`, which syncs text content over Socket.io events.

4.9 — Rich Text Editor (TipTap)

The textboard tab inside each board uses TipTap as a full-featured rich text editor. Users get bold, italic, underline, headings H1–H3, bullet lists, numbered lists, task lists (checkboxes), blockquotes, code blocks, horizontal rules, links, and highlighted text — all with a fixed formatting toolbar above the editor and a floating bubble toolbar on text selection.

TipTap is initialized via useEditor() with a suite of extensions: StarterKit (core formatting), Placeholder, Typography (smart quotes, dashes), Color + TextStyle (text color), Highlight (multi-color), TaskList + TaskItem (nested checkboxes), Image, and Link. The editor content is stored as TipTap JSON (a ProseMirror document tree), not HTML. JSON is safer to serialize, version, and diff than raw HTML strings.

Content syncs via a debounced approach: on every editor onUpdate, a 2-second debounce timer fires and calls socket.emit('text_update', { content: editor.getJSON() }). Other clients receive receive_update and call editor.commands.setContent(content, false) only if the editor is not currently focused — preventing the user's cursor from jumping while they type. Content also saves immediately to MongoDB on debounce.

Slash commands are implemented by listening for a / keydown event and rendering a floating dropdown. The dropdown filters as the user types (e.g., /h1, /todo, /code) and applies the chosen format via editor.chain().focus().... The @tiptap/extension-collaboration Yjs extension was explicitly excluded because CRDT adds binary protocol overhead and awareness complexity that is overkill for a "mostly one editor at a time" use case.

4.10 — Command Palette

A global command palette opens with Ctrl+K (or Cmd+K on Mac) from any page. It also opens by clicking the search bar in the sidebar. The backdrop fades in at 150ms, and the modal scales in from 0.96 with a fade, giving it a polished feel.

The palette searches all boards and whiteboards by title using simple substring matching: title.toLowerCase().includes(query.toLowerCase()). Results show the item's type icon, title, and last-updated timestamp. The user can navigate with arrow keys (activeIndex state), press Enter to open the selected board, or Escape to close.

The global keydown listener is attached in App.jsx via useEffect so it is always active regardless of which page is open. A proper fuzzy search library like Fuse.js was considered but rejected — substring matching is sufficient for the expected board count (tens to low hundreds), and it avoids an extra dependency.

4.11 — Soft Delete & Trash

Boards and whiteboards are never permanently deleted by a user action — they are soft-deleted. This means the document stays in MongoDB, but an isDeleted: true flag is set via PATCH /api/boards/:id/trash. The Trash page renders all soft-deleted items at 0.7 opacity with Restore and Permanently Delete options.

The key design decision was to compute all dashboard views (Recent, Starred, Trash, All) from the same MongoDB collections using different query filters — not separate collections. This keeps the data model simple: one Board collection, one Whiteboard collection, filtered views everywhere. The backend normalizes all responses to a unified shape { id, type, title, updatedAt, starred, isDeleted } so the frontend never needs to branch on data source.

The lastOpenedAt field is updated on every GET /api/boards/:id request, which drives the "Recent" view. This is a deliberate tradeoff: it adds a write on every read, but it's accurate and requires no separate tracking table.

4.12 — User Attribution Labels

Attribution labels are implemented for text objects in the standalone whiteboard. Each text object includes `authorName` so the label remains visible after the object is created.

Because the label is stored on the object itself, the UI does not need to infer authorship from a separate user map. This improves sync correctness and avoids stale attribution when user names change.

The board editor also includes typing indicators using Socket.io events such as `text_typing`. This is a lightweight collaboration signal rather than full per-object awareness.

SECTION 5 — Architecture Critique

5.1 — What is Good Architecture Here

  • Clear separation of concerns between synchronous REST routes and real-time socket events.
  • Shared overlay components for text, shapes, sticky notes, and images, which keep the code modular.
  • A simple relay-style Socket.io server that minimizes backend processing and keeps the collaboration model easy to reason about.
  • No premature optimization: the team avoided CRDTs, fuzzy search libraries, and heavyweight editor frameworks for the current MVP.
  • Single socket instance per client is implied by the architecture, reducing the risk of duplicate connections.

5.2 — What is Not Ideal Architecture

  • The in-memory `whiteboardParticipants` and `boardParticipants` maps are ephemeral. A restart drops presence data, which would be unacceptable in a production SaaS system.
  • Socket auth is lightweight and board-level only. Anyone with a valid board ID can connect, which means the system lacks robust user authentication.
  • Full-document sync for the board text editor is the current approach. That means concurrent edits can overwrite each other if two people type simultaneously.
  • Storing user identity in session/local storage means display names are not portable across devices and can be spoofed.
  • There is no socket-level rate limiting beyond a small connection-rate guard, so a malicious client could spam draw events and flood the room.

5.3 — Scalability Analysis

The current architecture is suitable for a small real-time collaboration app with up to a few hundred concurrent users across rooms. Socket.io can handle roughly 1,000 connections on a single server if the message rate stays moderate.

At 10,000 boards, MongoDB performance depends on indexes. The `boardId` and `whiteboardId` fields are indexed, so lookups remain fast. However, unbounded arrays of strokes and images will grow storage and query costs; sharded or archival strategies would be needed for a production-scale dataset.

For a production SaaS offering, the app would need Redis pub/sub for socket scaling, proper authentication, rate limiting, horizontal server scaling, a CDN for assets, and a more robust database partitioning strategy.

SECTION 6 — Difficulties & Decisions During Development

6.1 — The Coordinate System Bug

The symptom was that objects placed while the canvas was transformed would appear in the wrong spot after zoom or resize. The root cause was raw mouse coordinates being used without the normalizing transform.

The fix is to compute `canvasX = (mouseX - panX) / zoom` consistently in every placement and drag handler. The hard part is auditing every mouse event, because the mistake can hide in text placement, shape creation, and drag motion.

6.2 — Undo/Redo Sync Complexity

Undo is inherently hard with multiple clients because each client may have a different view of history. The simplified approach chosen here is to treat undo as a shared command and broadcast it to all clients.

This means if two people undo at the same time, the room may apply two undos and remove more strokes than intended. It is acceptable for the current scope because the use case is light collaboration rather than strict operational transform precision.

6.3 — The Whiteboard.jsx vs WhiteboardPage.jsx Divergence

There are two whiteboard implementations in the repo: `frontend/src/components/Whiteboard.jsx` and `frontend/src/pages/WhiteboardPage.jsx`. One is an embedded board used inside `BoardPage`, and the other is a standalone whiteboard page.

This divergence is dangerous because a bug fixed in one file may remain in the other. The long-term fix is to share core canvas logic via a common hook or shared component.

6.4 — Sticky Note Rotation Was Removed

The initial design included visually rotated sticky notes to mimic analog paper. Rotation was removed because CSS `transform: rotate()` interfered with coordinate math for drag, resize, and collision boundaries.

That taught an important lesson: visual polish that breaks the coordinate system must be deferred until the underlying geometry is stable.

6.5 — TipTap Extension Choices

The team seriously considered @tiptap/extension-collaboration and Yjs for the rich text editor. Yjs is a CRDT framework that enables true conflict-free concurrent editing. After evaluation, it was rejected because it introduces binary protocol encoding, an awareness layer, a separate Yjs document that must be synced alongside the React state, and significantly more complex setup — all for a feature (simultaneous editing in the same text block) that is rarely needed in this use case.

The chosen approach — debounced JSON sync every 2 seconds, applied only when the editor is not focused — handles the common case cleanly. Two users typing at the same moment may overwrite each other, but this is an acceptable tradeoff given the scope. The decision is clearly documented as a known limitation, not an oversight.

6.6 — Performance on Dense Whiteboards

With many strokes and overlay objects, rendering all objects on every update can lag. The fix is to redraw only when necessary and keep object overlays in separate components so the DOM does not re-render the entire board on every keystroke.

In React, `useMemo`, `useCallback`, and stable dependencies help reduce needless updates. The repo already separates canvas stroke rendering from DOM overlays, which supports better performance.

SECTION 7 — Viva Q&A Preparation

Category A — Fundamentals (10 questions)
  1. What is WebSocket vs HTTP? HTTP is request/response, while WebSocket is a persistent bidirectional channel. For BlackBoard, HTTPS fetches page data and REST state, while Socket.io uses websockets for live whiteboard updates.
  2. Why use Socket.io over raw WebSocket? Socket.io adds reconnection, room management, and event namespaces. These features simplify real-time collaboration without building them from scratch.
  3. What is a REST API? A REST API exposes data and actions over HTTP using resource-oriented routes. BlackBoard uses REST for board creation, unlocking, and admin operations.
  4. What is MongoDB? MongoDB is a document database that stores JSON-like objects. It fits BlackBoard because board documents contain flexible arrays of strokes and overlay objects.
  5. Why NoSQL for this project? NoSQL is a good fit because the whiteboard schema is heterogeneous and evolves quickly. It avoids rigid relational migrations for object arrays and embedded rich objects.
  6. What is React? React is a UI library for building component-based interfaces. It is used here to render pages, toolbars, and overlay components declaratively.
  7. What is a React hook? A hook is a function like `useState` or `useEffect` that manages component lifecycle or state. BlackBoard uses hooks extensively for canvas setup, socket effects, and local interaction state.
  8. What is useEffect? `useEffect` runs side effects after render. This repo uses it to initialize sockets, fetch data, and subscribe to window events.
  9. What is useState? `useState` holds local component state. It is used for tool selection, stroke arrays, and UI toggles.
  10. What is a component? A component is a reusable piece of UI with its own rendering logic. BlackBoard composes pages from `WhiteboardPage`, `TextObject`, `ShapeObject`, and others.
Category B — Architecture (10 questions)
  1. Why no Redux? The app is small enough to manage with local component state and prop drilling. Redux would add complexity without substantial benefit for the current feature set.
  2. Why no SQL? The data is document-oriented and flexible, so MongoDB is a better fit than a relational schema. SQL would require more rigid modeling for arrays of stroke objects and overlay metadata.
  3. How does real-time sync work? Client actions emit socket events to the server, which broadcasts them to the room. Other clients receive the event and update state accordingly.
  4. What happens if the server crashes? Active socket connections drop and clients reconnect. The backend persists board state periodically, so most progress is recovered from MongoDB after restart.
  5. How do you handle multiple users editing simultaneously? The app broadcasts edits in real time, but does not use a CRDT. Simultaneous edits can still overwrite each other, especially in the text editor.
  6. What is a Mongoose schema? It defines the shape of documents stored in MongoDB. `Board.js` and `Whiteboard.js` define fields, types, and defaults for persisted state.
  7. What is a socket room? A room is a logical group of connected sockets. BlackBoard uses rooms keyed by `boardId` or `whiteboardId` so only relevant clients receive events.
  8. Why is the socket server a relay? Because it forwards events with minimal processing. This keeps the architecture simple and avoids reimplementing the collaboration logic on the server.
  9. What is CORS and why do you need it? CORS controls which origins can access your API. The backend config allows the frontend origin so browser requests succeed from the deployed app.
  10. What is an environment variable? It is a configuration value available at runtime but not embedded in source code. BlackBoard uses them for VITE_API_URL on the frontend and MONGO_URI, JWT_SECRET, and CLIENT_URL on the backend.
Category C — Feature-Specific (10 questions)
  1. How does undo/redo work? Undo removes the last stroke from the `strokes` array and emits a socket event. Redo restores the last popped stroke from `redoStack`.
  2. How does zoom work? The canvas container gets transform: scale(zoom) translate(panX, panY) with transform-origin: 0 0. Every mouse coordinate is then normalized: canvasX = (mouseX - panX) / zoom. Zooming centered on the cursor uses a ratio calculation to keep the point under the cursor stationary.
  3. How do you know where to place an object on the canvas? You take the raw mouse position relative to the canvas bounding rect, then subtract the pan offset and divide by zoom: canvasX = (e.clientX - rect.left - panX) / zoom. This gives the stable canvas-space coordinate that persists regardless of zoom or pan.
  4. How do attribution labels work? When a text object is created, it records authorName, authorColor, and authorId from the user's session. The AttributionLabel component renders permanently above the text box using absolute positioning (top: -24px). Because the color is stored on the object itself, it remains accurate even after the original user leaves.
  5. How does the command palette search? A global Ctrl+K listener in App.jsx opens the CommandPalette component. All boards and whiteboards are filtered using title.toLowerCase().includes(query.toLowerCase()). Arrow keys change an activeIndex state; Enter navigates to the selected board via React Router.
  6. How does TipTap save content? TipTap's onUpdate callback fires on every edit. A 2-second debounce then calls socket.emit('text_update', { content: editor.getJSON() }) and saves the JSON to MongoDB. Other clients receive receive_update and apply it only if their editor is not focused, preventing cursor jumps.
  7. How does the QR code work? The ShareModal renders <QRCodeSVG value={boardUrl} size={160} /> from the qrcode.react library. The board URL is the full current window location. When scanned, the QR code navigates the scanner directly to the board — the QR encodes nothing more than a URL string.
  8. How do avatars get their colors? The server assigns colors deterministically by join order within the room. It counts existing users in roomUsers[boardId] and indexes into a 6-color palette. The assigned color is sent back to the client via the joined_board socket event and used for the avatar, attribution label, and cursor.
  9. How does soft delete work? A PATCH /api/boards/:id/trash request sets isDeleted: true on the document — it is never removed from MongoDB. The Trash page queries for isDeleted: true. The Restore action sets isDeleted: false. Permanent deletion calls a DELETE route that actually removes the document.
  10. How does the canvas grid render? CanvasGrid.jsx renders an SVG element absolutely positioned behind the canvas layer. It uses an SVG <pattern> element for dot or line grids, with the pattern offset adjusted by panX % spacing and panY % spacing so the grid appears to move with the canvas as you pan.
Category D — Security & Deployment (5 questions)
  1. How do you prevent exposing secrets? Keep secrets in backend environment variables and never bake them into frontend bundles. The frontend only uses public API endpoints.
  2. What is JWT used for? JWT authenticates socket connections and admin requests. The server verifies the token before allowing room membership or protected admin operations.
  3. What is the difference between Vercel and Render? Vercel is ideal for static frontends and serverless functions, while Render is better for long-lived backend services and websockets. BlackBoard would use Vercel for the React app and Render for the Express API.
  4. What is MongoDB Atlas? Atlas is a managed cloud database service for MongoDB. It hosts the database cluster and provides connection strings, backup, and access control.
  5. Why does your backend go to sleep? On free tiers, platforms like Render may cold-start inactive apps. This introduces latency for the first request after idle time. You can mitigate this by pinging the backend on a schedule or upgrading to a paid tier.
Category E — Critique & Improvement (5 questions)
  1. What would you do differently? I would separate socket logic into hooks, consolidate whiteboard components, and add a proper state layer for shared board context.
  2. How would you add proper authentication? Add user accounts, hashed passwords, session tokens, and access checks on every socket event. Use a shared identity service between the API and realtime server.
  3. What is a CRDT and when would you use one? A CRDT is a conflict-free replicated data type that enables concurrent edits without central locking. You would use it for true collaborative rich text or complex shared object state in high-collaboration scenarios.
  4. How would you scale this to 10,000 users? Use Redis pub/sub for socket rooms, shard the database, add horizontal backend instances, and separate static assets on a CDN.
  5. What is Redis and why might you need it? Redis is an in-memory datastore used for caching and pub/sub. You would need it when running multiple backend instances — Socket.io rooms exist in-process memory, so without Redis pub/sub, users on different server instances can't collaborate. It also helps cache frequently-accessed board data.

SECTION 10 — One-Page Cheat Sheet

5 Key Architectural Decisions

  • Socket.io relay-style sync — server forwards events without processing them, keeping the architecture stateless and simple.
  • Unified object system — every canvas object shares { id, type, x, y, width, height } so undo, sync, and rendering work identically for all types.
  • No Redux / Zustand — local state + localStorage is sufficient at this scale; avoids unnecessary boilerplate.
  • Debounced JSON sync for TipTap — 2-second debounce instead of Yjs/CRDT; simpler, reliable, good enough for one-editor-at-a-time usage.
  • MongoDB with unified API responses — all views (Recent, Starred, Trash, All) are computed from the same collections with filters, not separate collections.

5 Tricky Bugs & Fixes

  • Objects placed at wrong position when zoomed — fix: normalize every mouse coordinate with (mouseX - panX) / zoom in every single handler.
  • Undo desynced between tabs — fix: emit wb_undo socket event and apply the same state change on all clients.
  • Whiteboard.jsx vs WhiteboardPage.jsx divergence — fix: unify core canvas logic into a shared component/hook so bug fixes apply to both.
  • Sticky note rotation broke coordinate math — fix: remove rotation entirely; CSS rotate() transforms invalidate standard drag/resize math.
  • Stroke loss on fast draw + reload — fix: flush strokes to MongoDB immediately on mouse-up, not on a delayed batch interval.

5 Most Likely Viva Questions

  • Why MongoDB not SQL? — whiteboard objects are heterogeneous arrays; MongoDB's document model fits flexible, schema-light storage better than rigid relational tables.
  • Why no Redux? — the app has a small number of screens and simple cross-component interactions; local state is sufficient and keeps code simpler.
  • How does real-time sync work? — client emits a socket event → server relays to room → other clients update local state and re-render.
  • What is the coordinate normalization formula? — canvasX = (mouseX - panX) / zoom; must be applied everywhere a mouse position maps to canvas space.
  • How would you scale this? — Redis pub/sub for multi-instance socket rooms, horizontal backend scaling, MongoDB sharding, CDN for assets, and proper auth.
